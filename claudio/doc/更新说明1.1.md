# Claudio 最近更新说明

本文档汇总最近一轮与历史记录、页面展示、配置模板相关的更新。

## 1. 页面支持回填最近历史

之前 Claudio 的前端主要依赖 WebSocket 实时接收消息和播放事件，因此刷新页面后，聊天区和播放列表会重新变空，看起来像“数据丢了”。

现在已改为：

- 页面启动时会先请求后端历史接口
- 前端会从 SQLite 中回填最近的聊天记录和播放记录
- 回填完成后，再继续通过 WebSocket 接收新的实时事件

这意味着：

- 刷新页面后，不会再看到空白会话
- 页面会显示数据库中最近的一部分历史内容
- 数据是否存在，以 SQLite 为准，不再只看当前页面内存

## 2. 新增历史接口

后端新增了一个历史记录接口：

```http
GET /api/history
```

接口返回：

- 最近消息列表
- 最近播放列表

其中播放列表会尽量补出可播放 URL，方便前端直接恢复播放队列展示。

## 3. SQLite 自动限量清理

项目现在仍然使用 SQLite 保存播放记录和聊天记录，但不再无限增长。

已加入自动裁剪逻辑：

- 每次写入一条播放记录后，会检查并删除超出上限的旧播放记录
- 每次写入一条消息后，会检查并删除超出上限的旧消息记录

当前默认配置为：

```env
STATE_MAX_PLAYS=2500
STATE_MAX_MESSAGES=1000
```

说明：

- 播放记录最多保留 2500 条
- 聊天记录最多保留 1000 条
- 超出后自动删除更老的记录，只保留最近的数据

## 4. 页面历史展示上限可配置

页面启动时不会把数据库中全部历史都塞进前端，而是只回填最近一部分。

当前 `.env` 中支持以下参数：

```env
UI_HISTORY_PLAY_LIMIT=100
UI_HISTORY_MESSAGE_LIMIT=50
```

说明：

- `UI_HISTORY_PLAY_LIMIT`：页面刷新或新开页面时，最多回填多少条最近播放
- `UI_HISTORY_MESSAGE_LIMIT`：页面刷新或新开页面时，最多回填多少条最近消息

这两个参数只影响“页面初始化时显示多少”，不影响 SQLite 实际保留多少。

## 5. 当前页面行为说明

目前前端行为分成两段：

1. 页面启动时：
   - 先从数据库回填最近历史
   - 显示数量受 `UI_HISTORY_PLAY_LIMIT` 和 `UI_HISTORY_MESSAGE_LIMIT` 控制

2. 页面运行中：
   - 新消息和新播放会继续通过 WebSocket 实时追加到页面
   - 当前版本对“正在打开的页面内容”还没有再做单独裁剪

也就是说：

- 刷新后，页面只显示最近一部分历史
- 不刷新时，页面里的内容会继续增长
- 数据库本身不会无限增长，因为数据库有独立的保留上限

## 6. 新增 `.env.example` 模板

项目根目录已新增：

```text
.env.example
```

用途：

- 给新环境或新同学快速复制出 `.env`
- 避免把真实密钥和 Cookie 直接写进模板

模板中已保留：

- 必填配置项
- 可选配置项
- 数据库存储上限示例
- 页面历史展示上限示例

同时已将以下内容改为占位符：

- LLM API Key
- 天气 API Key
- 网易云 Cookie
- Fish Audio Key
- 飞书 Webhook

## 7. 推荐使用方式

如果你要调整历史相关行为，优先改 `.env`：

```env
STATE_MAX_PLAYS=2500
STATE_MAX_MESSAGES=1000
UI_HISTORY_PLAY_LIMIT=100
UI_HISTORY_MESSAGE_LIMIT=50
```

建议理解为两层：

- `STATE_MAX_*`：数据库层，决定最多保留多少条
- `UI_HISTORY_*`：页面层，决定刷新后首屏展示多少条

## 8. 注意事项

- 修改 `.env` 后，需要重启 Node 服务，配置才会生效
- 如果你希望“页面开着不刷新时也只保留最近 N 条”，还需要额外增加前端实时裁剪逻辑
- 当前版本已经解决“刷新后看不到历史”的问题，但没有限制运行中页面的实时追加长度

## 9. 本次更新适合解决的问题

这轮更新主要解决了下面几个实际痛点：

- 刷新页面后聊天和播放列表看起来像丢失
- 数据库长期运行会不断膨胀
- 配置项分散，难以统一调整
- 项目分享给别人时，缺少安全的环境变量模板
